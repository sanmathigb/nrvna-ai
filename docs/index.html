Submit now.
Collect later.

nrvna is a tiny async primitive for local AI workflows. Filesystem is the queue. Shell is the orchestrator. Three commands. No frameworks. No Python dependencies.

View on GitHub Get Started Patterns
quickstart
nrvnad model.gguf workspace        # load model, start processing
JOB=$(wrk workspace "Summarize this meeting transcript")
flw workspace -w $JOB              # wait and collect result
        wrk                    nrvnad                    flw
         │                        │                        │
 "prompt" ──▶ input/ready/ ──▶ processing/ ──▶ output/ ──▶ result
         │                        │                        │
     (submit)              (workers churn)            (collect)

Why This Exists

One-shot prompting is easy. Reliable async workflows are hard. nrvna keeps the core boring: atomic job transitions, inspectable queue state, and predictable CLI composition.

Atomic State Machine

input/ready → processing → output|failed via filesystem renames. No hidden queue broker.

Composable Primitives

Three commands: nrvnad, wrk, flw. Everything else is shell scripts.

Crash-Resilient

Jobs are directories. Daemon restarts recover orphaned work and continue processing.

Install

terminal
git clone --recursive https://github.com/sanmathigb/nrvna-ai.git
cd nrvna-ai
cmake -S . -B build && cmake --build build -j4
sudo cmake --install build

All dependencies are vendored. llama.cpp is built automatically — no separate install, no conflicts.

Get a Model

Download any GGUF model from HuggingFace and place it in ./models/:

terminal
mkdir -p models
# Download a model, e.g. Phi-3, Qwen2.5, Llama 3, etc.
# Place the .gguf file in ./models/
Or set NRVNA_MODELS_DIR to point to your existing models directory.

First Run

Start the daemon

Interactive mode shows a dashboard with your models and workspaces. Pick a number to start.

terminal 1
nrvnad                             # interactive mode
# or start directly:
nrvnad model.gguf workspace

Submit a job

terminal 2
$ wrk workspace "What is the capital of France?"
abc123

The job ID is printed immediately. The job processes in the background.

Collect the result

terminal 2
$ flw workspace abc123
The capital of France is Paris.

# Or use -w to wait for a job still processing:
$ flw workspace -w abc123

Piping

Submit and wait in one line:

terminal
wrk workspace "Hello" | xargs flw workspace -w
wrk prints a job ID to stdout. flw -w waits for the job to complete, then prints the result. Together they behave like a synchronous call.

Job States

Jobs are directories. State is location. Transitions are atomic renames.

StateDirectoryDescription
STAGINGinput/writing/<id>Being created, not yet visible to workers
QUEUEDinput/ready/<id>Waiting for a worker thread
RUNNINGprocessing/<id>Inference in progress
DONEoutput/<id>Completed successfully
FAILEDfailed/<id>Error occurred
input/writing/  ──rename──▶  input/ready/  ──rename──▶  processing/  ──rename──▶  output/
   (staging)                  (queued)                  (running)                 (done)
                                                            │
                                                            └──rename──▶  failed/
No database. No message broker. Inspect with ls, debug with cat, monitor with watch.

Workspace Layout

workspace/
├── input/
│   ├── writing/      ← jobs being staged
│   └── ready/        ← jobs queued for processing
├── processing/       ← jobs currently running inference
├── output/           ← completed results
│   └── <job-id>/
│       ├── prompt.txt
│       └── result.txt | embedding.json | audio.wav
└── failed/           ← errors
    └── <job-id>/
        ├── prompt.txt
        └── error.txt

Patterns composable

The primitives are small. What you build with them isn't.

Batch Processing

Submit many jobs, collect results later. Workers churn through them in parallel.

batch.sh
# Submit 100 image captioning jobs in seconds
for img in photos/*.jpg; do
  wrk workspace "Caption this image" --image "$img" >> jobs.txt
done

# Check progress
ls workspace/output/ | wc -l

# Collect all results
for job in $(cat jobs.txt); do
  echo "=== $job ==="
  flw workspace $job
done

Fan-Out / Fan-In

Break a task into parallel subtasks, then synthesize results.

fan-out.sh
# Fan-out: submit parallel subtasks
job1=$(wrk workspace "Research: PostgreSQL scalability")
job2=$(wrk workspace "Research: PostgreSQL ecosystem")
job3=$(wrk workspace "Research: PostgreSQL vs MongoDB")

# Wait for all
result1=$(flw workspace -w $job1)
result2=$(flw workspace -w $job2)
result3=$(flw workspace -w $job3)

# Fan-in: synthesize
wrk workspace "Synthesize these findings into a recommendation:
$result1
$result2
$result3"

Agent Loop

Iterate on a goal until done. Job history becomes agent memory.

agent-loop.sh
GOAL="Write a Python tutorial covering variables, loops, and functions"
memory=""

for i in {1..5}; do
  result=$(wrk workspace "Goal: $GOAL
Previous work: $memory
Continue. Write the next section. Say DONE if complete." | xargs flw workspace -w)

  echo "=== Iteration $i ==="
  echo "$result"

  if echo "$result" | grep -q "DONE"; then
    break
  fi

  memory="$memory\n---\n$result"
done

Self-Refinement

Generate, critique, improve.

refine.sh
GOAL="Write a cover letter for a senior engineer position"

# First draft
draft=$(wrk workspace "$GOAL" | xargs flw workspace -w)

# Critique
critique=$(wrk workspace "Critique this draft. What's weak? $draft" | xargs flw workspace -w)

# Improve
final=$(wrk workspace "Improve based on feedback:
Draft: $draft
Feedback: $critique" | xargs flw workspace -w)

echo "$final"

Multi-Model Routing

Different models for different tasks. Run multiple daemons, route work by type.

multi-model.sh
# Start specialized daemons
nrvnad qwen-vl.gguf    ws-vision &   # mmproj auto-detected
nrvnad codellama.gguf  ws-code   &
nrvnad phi-3-mini.gguf ws-fast   &
nrvnad outetts.gguf    ws-tts    &   # vocoder auto-detected

# Route by task type
wrk ws-vision "Describe this" --image photo.jpg
wrk ws-code   "Refactor: $(cat main.py)"
wrk ws-fast   "Classify: bug or feature?"
wrk ws-tts    "Welcome to nrvna" --tts

Memory / Context Accumulation

Build up knowledge across jobs. Job output IS context history.

memory.sh
# Save results to memory
flw workspace $job1 >> memory.txt
flw workspace $job2 >> memory.txt
flw workspace $job3 >> memory.txt

# Use accumulated context
wrk workspace "Given this context:
$(cat memory.txt)

Answer: What's the best approach for a startup?"

Embeddings

Generate vector representations for search and similarity.

embed.sh
# Generate embeddings for a corpus
for doc in docs/*.txt; do
  content=$(cat "$doc")
  job=$(wrk workspace "$content" --embed)
  echo "$doc $job" >> embed-jobs.txt
done

# Results: output/<job-id>/embedding.json
# Format: { "dim": N, "vector": [...] }

Text-to-Speech

Generate audio from text using OuteTTS models.

tts.sh
nrvnad outetts.gguf ws-tts       # vocoder auto-detected

job=$(wrk ws-tts "Welcome to nrvna ai" --tts)
flw ws-tts -w $job

# Result: workspace/output/<job-id>/audio.wav

Event-Driven

Monitor completions in real time with fswatch.

watch.sh
# Terminal 1: Watch for results
fswatch -0 workspace/output | while read -d '' path; do
  [[ "$path" == */result.txt ]] && cat "$path"
done

# Terminal 2: Submit jobs
wrk workspace "Question 1"
wrk workspace "Question 2"

Architecture

Directory-based job queue for LLM inference using llama.cpp. Jobs are filesystem directories that move through states via atomic renames.

Components

ComponentRole
WorkClient API — submits jobs, creates directories, writes prompts
FlowClient API — queries job status and retrieves results (thread-safe)
ServerOrchestrates Scanner + Pool + Processor lifecycle
ScannerFinds jobs in input/ready/ every 5 seconds
PoolThread pool for worker threads
ProcessorRoutes jobs by type, manages Runners, handles state transitions
RunnerWraps llama.cpp for text, vision, and embedding inference
TtsRunnerText-to-speech with OuteTTS + vocoder
LoggerThread-safe logging to stderr

Job Submission Flow

Client-side. wrk calls Work::submit().

1. Create directory:  input/writing/<job_id>/
         │
         ▼
2. Write prompt to:   input/writing/<job_id>/prompt.txt
   (+ type.txt for embed/tts, images/ for vision)
         │
         ▼
3. ATOMIC RENAME:     input/writing/<job_id>  →  input/ready/<job_id>
         │
         ▼
4. Return job_id to client

Job Processing Flow

Server-side. Scanner discovers jobs, Pool assigns workers, Processor runs inference.

1. Scanner finds job in input/ready/<job_id>
         │
         ▼
2. Pool assigns to worker thread
         │
         ▼
3. Processor::process():
   a. ATOMIC RENAME:  input/ready/<job_id>  →  processing/<job_id>
   b. Read prompt from processing/<job_id>/prompt.txt
   c. Read type from processing/<job_id>/type.txt (default: text)
   d. Route by type:
      • text/vision  →  Runner::run()     →  result.txt
      • embed        →  Runner::embed()   →  embedding.json
      • tts          →  TtsRunner::run()  →  audio.wav
   e. On success: write output, RENAME  →  output/<job_id>
   f. On failure: write error.txt, RENAME  →  failed/<job_id>

Result Retrieval

Client-side. flw calls Flow::status() and Flow::get().

Flow::status(job_id) checks directories in order:
  output/<job_id>       →  Status::Done
  failed/<job_id>       →  Status::Failed
  processing/<job_id>   →  Status::Running
  input/ready/<job_id>  →  Status::Queued
  none found            →  Status::Missing

Thread Model

Main Thread
    │
    ├── creates Server
    │       │
    │       ├── Scanner (1 thread)
    │       │     └── every 5s: scan input/ready/, submit to Pool
    │       │
    │       ├── Pool (N worker threads)
    │       │     ├── Worker-0  ──  Runner + TtsRunner
    │       │     ├── Worker-1  ──  Runner + TtsRunner
    │       │     └── Worker-N  ──  Runner + TtsRunner
    │       │
    │       ├── Processor (shared, thread-safe)
    │       │     ├── pre-initializes N Runners
    │       │     └── pre-initializes N TtsRunners (if vocoder present)
    │       │
    │       └── recoverOrphanedJobs (processing/ → ready/ or failed/)
    │
    └── waits for shutdown signal (SIGINT/SIGTERM)
  • Shared llama_model across all workers (thread-safe)
  • Per-worker llama_context created fresh for each job, freed after
  • Per-worker mtmd_context for vision (NOT thread-safe)
  • Vision encoding serialized via mutex (GGML shared compute graph state)

Inference Pipeline

Text / Vision

Based on llama.cpp examples/simple/simple.cpp and tools/mtmd/mtmd-cli.cpp.

  • Chat template applied via llama_chat_apply_template (falls back to raw prompt for base models)
  • Sampler chain: penalties → top_k → top_p → min_p → temp → dist
  • stripThinkBlocks() removes <think>...</think> from reasoning models

TTS

Based on llama.cpp tools/tts/tts.cpp.

  • OuteTTS v0.2/v0.3 auto-detected by vocabulary probing
  • Audio code generation with top_k=4 sampler
  • Code extraction via <|N|> token text parsing
  • Vocoder: codes → embeddings → ISTFT spectral conversion → 24kHz PCM

Embeddings

  • Creates context with embeddings=true, mean pooling
  • Returns float vector (dimension depends on model)
  • Output: JSON with dim and vector fields

CLI: wrk

wrk --help
Usage: wrk <workspace> <prompt...> [--image <path> ...]
       wrk <workspace> <text> --embed
       wrk <workspace> <text> --tts
       wrk <workspace> -                 (read prompt from stdin)

Arguments:
  workspace     Directory for job storage
  prompt        Text prompt (can be multiple words)
  -             Read prompt from stdin

Options:
  --image <path>   Attach image (repeatable)
  --embed          Submit as embedding job
  --tts            Submit as text-to-speech job
  -h, --help       Show help
  -v, --version    Show version

CLI: flw

flw --help
Usage: flw <workspace> [options] [job_id]

Arguments:
  workspace     Directory for job storage
  job_id        Specific job ID (optional)

Options:
  -w, --wait    Wait for job to complete
  -h, --help    Show help
  -v, --version Show version

Behavior:
  job_id given  →  retrieve that job
  no job_id     →  retrieve latest completed job
  stdin         →  read job_id from pipe
